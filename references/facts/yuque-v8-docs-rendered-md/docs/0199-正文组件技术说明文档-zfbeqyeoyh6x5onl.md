# 一、项目概述

## 1.1 项目背景

@seeyon/biz-content 作为 PC 端核心正文组件已运行多年。业务膨胀、信创差异、客开定制激增，原实现暴露出以下痛点：

○
性能：重复渲染频繁，初始化全量加载，耗时长，主线程阻塞；

○
维护：业务逻辑高度耦合，客开代码靠标识硬编码；

○
扩展：工具栏、控件动作、三方渠道无统一替换入口，客开无标准方案。

为解决上述问题，项目启动“正文组件运行时级重构”，新代码具备以下特点：

●
物理隔离：新逻辑置于 packages/content/src/runtime，与优化前的老代码互不干扰，相互隔离；

●
状态切换：cmi 企业、客户端插件企业、非金山/永中通道仍走老逻辑，其余自动进入优化后的代码；

●
插件扩展：工具栏、控件动作、原子通道提供统一插槽，客开零改动标品即可注入定制代码；

●
性能提升：精简刷新、懒加载减包，业务与渲染层解耦；

●
维护简化：各业务模块独立，耦合度显著下降。

## 1.2 适用版本

@seeyon/biz-content 5.30 以后的版本（2026.3.30以后的版本）

# 二、架构设计

目前正文组件针对标准正文（富文本），金山通道的版式，流式文件，默认采用的是重构之后的代码。针对其余的能力通道(包含特殊的客户场景代码，如非cmi租户，使用客户端插件的用户)，则仍然使用重构之前的业务代码。

## 2.1 模块架构图

根据正文的业务特点，将正文组件业务分为了以下几大模块

●
状态管理 BizContentContext 负责正文的全局状态维护，工具栏，控件动作等业务函数的入参均为context

●
工具栏    BizContentToobar 负责渲染正文的工具栏，目前提供了按钮式，文字式，悬浮式等渲染模式

●
业务逻辑 BizContentActions 统一维护工具栏，控件动作的业务函数，不同业务之间相互隔离。

●
手写调用 BizContentExport  负责维护正文向外暴露的组件，如contentReader,contentWriter等

●
渲染层    BizContentRenderer 根据context中维护的通道能力，对接三方渲染机制。不同通道之间相互隔离。

●
预渲染层 BizContentPreviwer  在只读预览态，为了降低初始化白屏时间，标准正文直接通过dangerouslySetInnerHTML进行渲染，非标准正文则通过图片进行预加载渲染。（TODO-20260630实现）

模块之间，模块内部均采用宁冗余，不耦合的维护原则，可以抽取公共方法，但是切勿将不同的业务逻辑进行耦合。下图为正文组件的模块架构示例图。

# 五、套红与智能排版

套红与智能排版是依托专用模板，借助模板内置书签定位，自动将业务文单字段填充至模板指定位置，快速完成红头、红章的标准化版式制作的业务流程。目前正文组件中，如果存在智能排版权限，则不能进行套红操作。二者核心差异及二次处理规则：

首次操作区别：套红直接将原文内容填入模板的正文书签；智能排版需调用第三方智能服务(目前是依赖方寸服务)先对原文内容进行智能调整，智能调整完成后再填入正文书签位置。

二次操作区别：二次套红以首次套红模板为基准，仅更新书签内容；而对已智能排版的文档再次排版时，系统会提取上一轮排版后的正文作为源内容，重新进行智能处理后再回填书签。

目前套红，智能排版，更新书签相关业务，暂只支持公文业务中的word文字类型的文档。

关键步骤讲解

套红/更新书签逻辑判断：这一步，主要需要为 更新书签，套红方法做准备，需要确定出三个参数：套红内容的storeKey，套红文件的storageKey，是否为更新书签操作。从上到下主要分为四种情形

上次套红模板等于本次套红模板：代表更新书签操作

●
其它 ：套红内容的storeKey来源于 hualian对象中的 original数据

处理花脸数据：更新书签的时候不执行这一步骤。一次套红需要存储  original  数据，二次套红会将当前数据存为花脸数据

更新书签、套红：根据套红内容的storeKey，套红文件的storageKey，是否为更新书签操作做执行获取书签信息，调用接口进行套红。这儿需要注意以下几点：

●
  如果是套红操作中的更新书签，则调用接口中的数据不传正文书签（bookmark: 'Content'）；智能模板默认是需要传的；

●
  接口中的套红文件templateStorageKey：如果是套红中的更新书签，则传递的是当前文件的storeKey。其余情况均是套红模板的key.

**更新意见：**将文单上的意见数据利用sdk提供的插入能力更新到正文意见书签中。书签插入的是一个复杂的html数据，这一块是前端通过金山提供的sdk插入的。

**处理版记偶数页：**判断生成的版记位置，如果在奇数页，则让版记新建一个页面，版记下边缘与偶数页的版心下边缘重合，如果在偶数页则按套红模板生成即可，不做调整。套红模板设置中修改配置项后，不影响已经套红后的正文。

**缓存文单数据：**调用后端接口缓存当前的文单数据，下次更新书签就以当前更新的文单数据作为比较基准。

**更新单一记录：**将最新的正文数据更新到单一记录中去。

# 六、签章

目前公文业务中的签章分为两大类，一类是由三方签章通过提供的电子签章能力，另外一种是公文内部提供的图片章能力，通过三方sdk插入文档中。三方电子签章后，会在正文的值中存放对应的签章后的文件，和已签章的标识。图片章则是基于当前文件进行操作，无法判断是否已完成签章。
