# 泛化调用

> Source: https://www.yuque.com/seeyonkk/v8/cyxtucgh7gn9gggc

作者：李阳CD

最后更新：2025-04-11

适用版本：V3.5

# 1
应用场景

泛化调用是利用Java反射和Dubbo动态代理的机制，跨应用或服务调用AppService方法。泛化调用使用场景非常多，比如跨应用获取枚举值、跨应用获取应用参数、应用间依赖解耦、调用平台标准应用对外的AppService方法，复用平台的能力等。

1
场景一：跨应用获取枚举值。

2
场景二：跨应用获取应用参数。

3
场景三：应用间依赖解耦。

4
场景四：查询UDC自建应用的实体数据。

必须满足以下几个条件才能支持泛化调用：

1
被调用方的Service类必须添加了@AppService注解。

```java
package com.seeyon.boot.starter.parameter.appservice;

@AppService //被调用方必须有这个注解
public class CtpParameterAppService {
    
    @Autowired
    private CtpParameterService service;
    
    public SingleResponse<Map<String, Object>> selectValuesByApp(SingleRequest<String> request) {
        return SingleResponse.from(this.service.selectValuesByApp((String)request.getData()));
    }
}
```

2
明确被调用方注册的服务名称。

如果被调用方是平台标准应用，注册的服务名称一般是“关于”页面的“应用标识”字段。

3

泛化调用同反射调用一样，需要知道被调用方的包路径、方法名称、入参和出参的数据类型等，参数类型不一致会导致泛化调用失败，可通过反编译或ctp-studio平台拉取应用源码的方式来明确这些信息。

# 3
实现步骤

## 3.1
创建自定义微流程

在调用方应用中，实现一个自定义微流程，后续在自定义微流程中去编写泛化调用代码。

实现自定义微流程详见“自定义微流程”章节。

## 3.2
泛化调用参数及流程说明

泛化调用主要使用Java反射和Dubbo动态代理机制，构造好调用参数调用对应方法即可。

# 4
场景示例

## 4.1
场景一：跨应用获取枚举数据

V8的枚举数据没有统一存储到公共数据表，而是每个应用的枚举单独存放于自己的应用数据库中，这就导致常规情况下应用只能获取自己的枚举，而使用泛化调用，即可跨应用查询其他应用的枚举数据。

## 4.2
场景二：跨应用获取应用参数

UDC自建应用的应用参数没有统一存储到公共数据表，而是每个应用的参数单独存放于自己的应用数据库中，这就导致常规情况下应用只能获取自己的参数，而使用泛化调用，即可跨应用查询其他应用的参数数据。

## 4.3
场景三：应用间依赖解耦

使用Dubbo远程调用的方式，虽然可以在代码中直接调用service方法，但必须要加入到应用依赖。

在某些复杂业务的情况下会产生应用间循环依赖，这是平台不允许的，这就会导致Dubbo远程调用的方式不可用。

比如这样的依赖和调用方式不被允许：应用A->应用B->应用C->应用D->应用A

而泛化调用不需要加入应用依赖即可调用，所以可以使用此特性对应用进行依赖解耦。

以上方循环依赖为例，应用D可以使用泛化调用的方式去调用应用A提供的AppService方法，即可达到解耦的目的，具体代码不再阐述。

## 4.4
场景四：查询UDC自建应用的实体数据

当应用不能循环依赖时，可以通过泛化调用的方式跨应用查询实体数据。

每个应用的实体编码可以使用平台标准方法动态查询，此处不再赘述，只演示固定查询某个实体的数据。

# 5
注意事项

●
如果不清楚目标方法的执行原理和逻辑，请谨慎使用泛化调用，避免出现不可预知的问题甚至宕机风险。

●
不同的V8平台版本对应的泛化调用方式可能存在差异，示例代码仅供参考。
