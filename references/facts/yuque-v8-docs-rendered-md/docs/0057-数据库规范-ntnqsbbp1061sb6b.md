# 数据库规范

> Source: https://www.yuque.com/seeyonkk/v8/ntnqsbbp1061sb6b

## 数据库规范

### 命名规范

1
采用下划线_命名法。

2
必须使用小写字母或数字，禁止出现数字开头，禁止两个下划线中间只出现数字。

3
不超过 30 个字符。

4
禁用保留字，请参考官方保留字。

5
常用名称参考术语表，例如：组织unit、人员member。

6
所有存储相同数据的列名和列类型必须一致，比如user表中的user_id和order表中的user_id。

7
临时表必须以tmp_为前缀并以日期为后缀。

8
备份表必须以bak_为前缀并以日期为后缀。

#### 表命名

●
表名不使用复数名词。说明：表名应该仅仅表示表里面的实体内容，不应该表示实体数量，对应于 java entity 类名也是单数形式，符合表达习惯。

●
表命名的规则分为 2 个层级，层级之间通过下划线_分割，例如org_member。规约为：

```
[app]_[name]
```

[app] 应用名称。

[name] 表的作用，可再细化层级。

#### 字段命名

●
表达是与否概念的字段，必须使用 is_xxx 的方式命名，数据类型是 tinyint（1 表示是，0 表示否）。对应于 java entity 中不要加is。例如：表达逻辑删除的字段名is_deleted，1 表示删除，0 表示未删除。

#### 索引命名

●
主键索引名自动生成；唯一索引名为uk_表名_字段名；普通索引名则为idx_表名_字段名。

### 设计规范

1
【强制】所有的表必须包含 id、tenant_id、create_time、update_time、version 字段，对应于 java entity 继承BaseEntity即可。

○
id：逻辑主键，用于对行数据的唯一性进行标识。 类型：bigint。所有的表必须创建逻辑主键，采用统一的主键便于分库分表以及数据抽取。要求是全局唯一，通过SnowFlake 算法获取。

○
tenant_id：标记租户信息。类型：varchar(20)。

○
create_time：标记数据创建时间。类型：datetime。

○
update_time：标记数据最后修改时间，用于数据增量 ETL 或缺陷回溯。类型：datetime。

○
version：版本号，用于乐观锁支持。类型：smallint。

2
【强制】禁止使用存储过程、视图、触发器、Event。说明：存储过程难以调试和扩展，更没有移植性。

3
【强制】禁止存储大文件或者大照片，非结构化数据请使用文件存储。

4
【强制】数据存储不允许使用 XML，避免序列化和反序列化导致的性能问题。如果必须存储复合数据，建议使用 JSON。

5
【强制】禁止使用外键，如果有外键完整性约束，需要应用程序控制。说明：外键会导致表与表之间耦合，update 与 delete 操作都会涉及相关联的表，十分影响 sql 的性能，甚至会造成死锁。高并发情况下容易造成数据库性能，大数据高并发业务场景数据库使用以性能优先。

6
【强制】小数类型为 decimal，禁止使用 float 和 double。说明：float 和 double 在存储的时候，存在精度损失的问题，很可能在值的比较时，得到不正确的结果。如果存储的数据范围超过 decimal 的范围，建议将数据拆成整数和小数分开存储。

7
【强制】所有的时间字段均以时间戳（Java 十三位标准）的方式存储，Mysql 对应datetime类型。

8
【强制】varchar 是可变长字符串，不预先分配存储空间，长度不要超过 4000，如果存储长度大于此值，定义字段类型为 longtext，可独立出来一张表，用主键来对应，避免影响其它字段索引效率。

9
【强制】必须把字段定义为 NOT NULL，varchar、longtext类型除外。

○
null 的列使索引/索引统计/值比较都更加复杂，对 MySQL 来说更难优化；

○
null 这种类型 MySQL 内部需要进行特殊处理，增加数据库处理记录的复杂性；同等条件下，表中有较多空字段的时候，数据库的处理性能会降低很多；

○
null 值需要更多的存储空间，无论是表还是索引中每行中的 null 的列都需要额外的空间来标识；

○
对 null 的处理时候，只能采用 is null 或 is not null，而不能采用=、!=、in、not in、<、> 这些操作符号。如：where name!=’shenjian’，如果存在 name 为 null 值的记录，查询结果就不会包含 name 为 null 值的记录；

如果使用默认值，在代码中处理，不要使用数据库default约束，以下是业务确定不了的情况下的默认值，如果业务能确定就使用业务确定的默认值：

| 实体字段类型 | 数据库字段类型 | 默认值 |
| --- | --- | --- |
| 文本、多行文本 | varchar | null |
| 整数 | int | 0 |
| 长整数 | bigint | 0 |
| 小数 | decimal | 0 |
| 日期、日期时间 | datetime | 1970-01-01 08:00:00 |
| 布尔 | tinyint | 0 |
| java枚举 | smallint | 0 |
| 自定义枚举（枚举组件） | varchar | null |
| 实体 | bigint | -1 |

10
【推荐】字段允许适当冗余，以提高查询性能，但必须考虑数据一致。冗余字段应遵循：不是频繁修改的字段。不是 varchar 超长字段，更不能是 longtext 字段。

11
【推荐】如果修改字段含义或对字段表示的状态追加时，需要及时更新字段注释。

12
【参考】合适的字符存储长度，不但节约数据库表空间、节约索引存储，更重要的是提升检索速度。例如：

| 字段名称 | 中文名称 | 字段类型 | 字段长度 |
| --- | --- | --- | --- |

| update_time | 修改时间**【固定字段】** | datetime |  |
| version | 版本号，用于乐观锁支持**【固定字段】** | smallint |  |
| name | 名称 | varchar | 255 |
| title | 标题 | varchar | 1000 |
| create_user_id | 创建用户 | bigint |  |

### 索引规范

1
【强制】索引字段的长度不能超过767bytes，大概是191个字符，组合索引也是。

2
【强制】超过三个表禁止 join。需要 join 的字段，数据类型必须绝对一致；多表关联查询时，保证被关联的字段有索引。说明：即使双表 join 也要注意表索引、SQL 性能。

3
【强制】页面搜索严禁左模糊或者全模糊，如果需要请走搜索引擎来解决。说明：索引文件具有 B-Tree 的最左前缀匹配特性，如果左边的值未确定，那么无法使用此索引。

4
【强制】防止因字段类型不同造成的隐式转换，导致索引失效。

5
【强制】禁止在更新十分频繁、区分度不高的属性上建立索引。说明：a）更新会变更 B+树，更新频繁的字段建立索引会大大降低数据库性能 b）“性别”这种区分度不大的属性，建立索引是没有什么意义的，不能有效过滤数据，性能与全表扫描类似。

6
【推荐】如果有 order by 的场景，请注意利用索引的有序性。order by 最后的字段是组合索引的一部分，并且放在索引组合顺序的最后，避免出现 file_sort 的情况，影响查询性能。正例：where a=? and b=? order by c; 索引：a_b_c 。反例：索引中有范围查找，那么索引有序性无法利用，如：WHERE a>10 ORDER BY b; 索引 a_b 无法排序。

7
【推荐】建组合索引的时候，区分度最高的在最左边。正例：如果 where a=? and b=? ，a 列的几乎接近于唯一值，那么只需要单建 idx_a 索引即可。说明：存在非等号和等号混合判断条件时，在建索引时，请把等号条件的列前置。如：where a>? and b=? 那么即使 a 的区分度更高，也必须把 b 放在索引的最前列。

8
【推荐】单表索引建议控制在 5 个以内。

9
【推荐】单索引字段数不允许超过 5 个。说明：字段超过 5 个时，实际已经起不到有效过滤数据的作用了。

10
【参考】创建索引时避免有如下极端误解： 1）宁滥勿缺。认为一个查询就需要建一个索引。 2）宁缺勿滥。认为索引会消耗空间、严重拖慢更新和新增速度。 3）抵制惟一索引。认为业务的惟一性一律需要在应用层通过“先查后插”方式解决。

### SQL 规范

1
【强制】禁止三表（含）以上联查，容易导致性能问题。

2
【强制】禁止大表使用 JOIN 查询，禁止大表使用子查询。说明：会产生临时表，消耗较多内存与 CPU，极大影响数据库性能。

3
【强制】禁止不分页查询所有数据。

4
【强制】禁止列表查询大字段。

5
【强制】禁止按大字段查询。

6
【强制】 在代码中写分页查询逻辑时，若 count 为 0 应直接返回，避免执行后面的分页语句。

7
【强制】禁止在 WHERE 条件的属性上使用函数或者表达式。

8
【推荐】在表查询中，一律不要使用 * 作为查询的字段列表，需要哪些字段必须明确写明。说明：1）增加查询分析器解析成本。2）增减字段容易与 resultMap 配置不一致。

9
【推荐】in 操作能避免则避免，若实在避免不了，需要仔细评估 in 后边的集合元素数量，控制在 1000 个之内。
