# 同步模式

选择增量同步条件：如果三方系统数据能提供停用数据，一律使用增量同步。

选择全量同步条件：三方只能提供所有并启用的数据，无法提供停用数据（比如某人离职）；那么V8会进行全量数据比对，中间表不存在的数据会进行停用，此方式有巨大危险，比如中间表只有一条数据，你也用全量，那么会将系统其他所有数据停用，造成生产事故，务必谨慎！

# 同步维度

可以根据需要灵活设定同步范围，对于项目初始化数据来讲，可以逐步单项从上而下同步，避免过多的报错，先同步组织，组织没问题了再同步岗位，……，最后同步人员。

对于一个完整组织同步来讲，组织、岗位、职级、人员是必需的，职务非必需。

同步顺序：因为组织扩展字段可能包含了人员，比如：组织的负责人，所以扩展字段在最后同步

组织→职级→岗位→职务→人员及任职→人员扩展字段→组织扩展字段。

# 中间表字段定义

中间表都在cip-connector数据库中，分为以下几个维度。

## 1、组织：cip_sync_unit表

包含了内部和外部的机构、部门

## 2、岗位 cip_sync_post表

## 3、职务 cip_sync_duty表

## 4、职级 cip_sync_level表

## 5、人员 cip_sync_member表

# 公共重要字段

third_id：三方id，三方唯一标识，根据该字段更新v8的记录行（包括code）。常和code混淆，如果确定三方code唯一不会变，那么可以code=third_id，降低理解成本。

third_update_time：三方更新时间，假设同一个组织或者人员等，往中间表插入了多次后再执行同步，那么取该字段最新的一条记录进行同步，其他记录忽略，如果third_update_time值都一样，那么随机取一条。

batch和source_batch：batch是批次号，source_batch是来源批次号。批次号是一个时间点，某时间执行了同步任务，那么该时间同步的数据的batch就是该时间。

一条数据因为重试次数设置，可以被同步多次，那么每次同步，batch都会变化，但是source_batch就是第一次同步的batch，不会改变，方便知晓该数据是什么时候第一次被同步的。

str1~str10:扩展字段，跟设定有关,如果扩展字段是人，那么传人员在v8中的code

# 数据demo及注意事项

## 组织

parent_code：上级的thirid_id,(因开始是按code模式同步，兼容历史原因，该字段名没做改变)，只有组织维度按照third_id确定的上下级，其他模块都是按照code确定所属关系

停用：如果上级A单位传的停用，下级B单位启用，那么最终A下属所有组织（包括B）和人员都会被停用。

# 同步数据查询条件

●
每次同步数据查询条件：status in(0,1) and retry<={重试次数}
status=0(初始状态)、 status=1(失败状态)

每同步一次，有一个批次号，代表该批同步的数据。source_batch代表原始批次号。

<img width="912.5">

第1次中间表没数据，2026-01-01 00:00:00 同步单位1，失败。
